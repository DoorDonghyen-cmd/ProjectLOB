# `/work` — 작업 세션 표준 워크플로우

> 유저가 `/work` 또는 "작업 시작", "다음 할 일" 등으로 세션을 시작할 때 이 절차를 따른다.

---

## Phase 1: 브리핑 (🔍 상황 파악)

**목적**: 현재 프로젝트 상태를 파악하고, 오늘의 작업 목표를 확정한다.

1. `00-project-manager/task_tracker.md`를 읽는다.
   - `doing` 섹션에 `[x]` 완료 항목이 있으면 → `complete`로 이동 제안
   - `doing` 섹션에 미완료 항목이 있으면 → 이어서 진행할지 확인
   - `doing` 섹션이 비어 있으면 → `track`에서 다음 후보를 브리핑
2. `05-bug-report/reports/bug_tracker.md`를 읽는다.
   - 활성 이슈가 있으면 유저에게 알린다.
   - task_tracker 버그 로그와 상태가 불일치하면 동기화한다.
3. 유저에게 **오늘의 목표**를 간결하게 브리핑하고 확정을 받는다.

> [!IMPORTANT]
> 브리핑 없이 바로 코드 작업에 들어가지 않는다. 유저가 명시적으로 "바로 코드 작성해"라고 지시한 경우에만 이 단계를 축약한다.

---

## Phase 2: 기획 (📐 설계 & 계획)

**목적**: 확정된 목표에 대해 설계 및 개발 계획을 수립한다.

1. 관련 기획/설계 문서를 확인한다 (프로젝트 내 기획 관련 문서 혹은 스킬 문서 참조).
2. 설계/기획이 충분하면 → **구현 계획(implementation_plan.md)** 아티팩트를 작성한다.
3. 새로운 시스템이거나 기존 설계와 상충되면 → 기획안을 먼저 작성하고 유저 확인을 받는다.
4. 구현 계획에 대해 **유저 승인**을 받은 후 Phase 3로 넘어간다.

> [!TIP]
> 단순 버그 수정이나 소규모 변경은 구현 계획 없이 바로 Phase 3로 넘어가도 된다. 판단 기준: "파일 3개 이상 수정" 또는 "새로운 클래스 생성"이면 계획 작성.

---

## Phase 3: 실행 (⚙️ 코드 작성 & 테스트)

**목적**: 승인된 계획에 따라 실제 코드를 작성하고 검증한다.

1. `task.md` 아티팩트를 생성하여 세부 체크리스트를 관리한다.
2. 코드를 작성한다.
   - 기존 코드 패턴과 아키텍처를 준수한다.
   - 요구 사항에 명시된 수치/사양을 정확히 반영한다.
3. 빌드 및 테스트:
   - 컴파일 에러 확인 및 유효성 검증
4. 작업 중 버그를 발견하면 `05-bug-report` 절차에 따라 즉시 기록한다.

---

## Phase 4: 마무리 (📝 기록 & 정리)

**목적**: 이번 세션의 작업 내역을 빠짐없이 기록하고, 프로젝트 상태를 갱신한다.

> [!IMPORTANT]
> 이 단계는 **절대 스킵하지 않는다**. 유저가 "정리해 줘", "마무리", "끝" 등을 말하거나, 작업이 자연스럽게 종료되었을 때 반드시 수행한다.

### 4-1. 태스크 트래커 갱신
- `00-project-manager/task_tracker.md`에서:
  - 완료된 항목을 `doing` → `complete`로 이동
  - 새로 시작한 항목을 `track` → `doing`으로 이동
  - `최근 작업 기록` 테이블에 오늘 날짜와 작업 내용 추가

### 4-2. 데일리 로그 작성
- `04-daily-logs/SKILL.md`에 오늘 날짜 섹션을 추가:
  - 🎯 목표
  - 🛠️ 개발 내역
  - 📁 수정된 주요 파일
  - 💡 다음 예정 작업

### 4-3. 히스토리 아카이브
- `06-history-archive` 스킬 절차에 따라:
  - `walkthrough.md` 생성 시 → `walkthrough_master.md`에 등록
  - `implementation_plan.md` 생성 시 → `implementation_plan_master.md`에 등록

### 4-4. 버그 트래커 동기화
- 이번 세션에서 발견/해결한 버그가 있으면:
  - `05-bug-report/reports/bug_tracker.md` 갱신
  - `task_tracker.md`의 버그 로그 테이블도 동기화

## 축약 모드

유저가 빠른 작업을 원할 때 (예: "간단한 수정", "이것만 고쳐 줘"):

- Phase 1: 생략 가능 (유저가 목표를 이미 명시)
- Phase 2: 생략 가능 (소규모 변경)
- Phase 3: 수행
- Phase 4: **항상 수행** — 최소한 태스크 트래커와 데일리 로그는 갱신
