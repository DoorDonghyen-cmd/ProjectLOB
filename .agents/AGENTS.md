# Project-Scoped Rules

## 리팩토링 진단 규칙 (Refactoring Diagnosis Rule)
- **트리거(Trigger):** 사용자가 "코드 리팩토링 필요할까?" (또는 유사한 뉘앙스)라고 질문할 때.
- **행동(Action):**
  1. **코드 상태 분석:** 즉시 현재 포커스 된 파일이나 관련된 주요 시스템 파일들의 상태(파일 크기, 줄 수, God Object 여부, 결합도, 단일 책임 원칙 위배 등)를 분석합니다. (예: `view_file` 또는 `grep_search`를 통한 구조 파악)
  2. **진단 보고:** 현재 코드가 리팩토링이 필요한 상태인지, 아니면 당분간 기능 구현에 집중해도 괜찮은 상태인지 객관적인 피드백을 제공합니다.
  3. **방향 제안:** 리팩토링이 필요하다면 어떤 모듈을 어떻게 쪼개는 것이 좋은지 구체적인 방향(예: UI 컴포넌트화, 로직 매니저 분리 등)과 현재 시점에서 진행할지 미룰지에 대한 '타이밍'을 함께 제안합니다.

## 작업 종료 자동 깃 서밋 규칙 (Session-End Auto Git Push Rule)
- **트리거(Trigger):** 사용자가 "오늘 작업은 여기까지 할게", "작업 끝", "퇴근" 또는 유사하게 세션을 정리하겠다는 의사를 표현할 때.
- **행동(Action):**
  1. **깃 상태 분석:** 즉시 `git status`를 실행하여 작업 트리에 커밋되지 않은 변경 사항이 있는지 확인합니다.
  2. **자동 커밋 및 푸시:** 변경 사항이 있는 경우, 오늘 변경한 파일과 주요 수정 내역을 요약하여 직관적인 커밋 메시지를 자동 생성하고 커밋한 뒤, `git push origin main`을 실행하여 원격 저장소에 자동으로 코드를 업로드합니다.
  3. **결과 보고:** 업로드된 내역(파일 목록 및 커밋 메시지)과 깃 링크를 보여주며 세션을 종료합니다.
