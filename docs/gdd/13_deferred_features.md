## 보류 기준

아래 중 하나 이상에 해당하면 이 문서로 이관한다:
1. **코어 재미와의 직결성이 약하다** — 있으면 좋지만 없어도 코어 루프가 동작한다.
2. **현 단계의 복잡도 상한을 넘긴다** — v1 범위에서 쌓으면 설계가 무거워진다.
3. **선행 시스템이 아직 없다** — 의존하는 기반이 먼저 완성되어야 착수 가능하다.

## 1. 투영(Projection) UI

> **출처**: [11_core_mechanism_intention.md §8](file:///d:/ProjectLoB/docs/gdd/11_core_mechanism_intention.md)

### 설계 요약
- 발사 전 결과 예측 UI. 적의 현재 위치(실체 토큰)와 턴 종료 후 예상 위치(고스트 토큰)를 같은 트랙에 동시 표시.
- 스택 발사 계획 시 실시간으로 고스트 갱신: 죽을 적 소멸, 생존 적 전진 위치 + 잔여 HP.
- "미리보기 후 확정" 2탭 구조.

### 보류 사유
- 현 단계에서는 거리 표시 라벨과 전투 로그만으로 충분히 상황 판단이 가능하다.
- 다수 적 시스템이 완성된 후에야 본격적으로 가치를 발휘할 기능이므로, 다수 적 구현 이후 재검토.
- 모바일 작은 화면에서의 정보 밀도 압축 설계가 별도로 필요하며, UI/UX 전문 검토가 선행되어야 한다.

### 복기 트리거
- 다수 적 공유 트랙이 동작하고, 플레이어가 "이 순서로 쏘면 어떻게 되는지 알 수 없다"는 피드백이 나오면 착수.

## 2. 발사방식 성장 축 (3연발/연발 등)

> **출처**: [11_core_mechanism_intention.md §5](file:///d:/ProjectLoB/docs/gdd/11_core_mechanism_intention.md) — *"v1 미설계, 다음 단계"*

### 설계 요약
- 세 성장 축 중 마지막. 스택의 소비 패턴(한 번에 끌어오는 양·방식)을 바꾸는 손잡이.
- 예: 3연발 = 위 3개를 합쳐서 한 발로 발사 → 적재 순서의 중요도가 더 올라감.

### 보류 사유
- 문서 자체가 v1 미설계로 명시. 스택 소비 패턴을 건드리므로 코어에 가장 민감한 축.
- 기본 LIFO 단발 사격의 재미가 충분히 검증된 이후에 도입해야 설계 리스크가 줄어든다.

## 3. 탄창 성장 (변형 아키타입)

> **출처**: [11_core_mechanism_intention.md §5](file:///d:/ProjectLoB/docs/gdd/11_core_mechanism_intention.md)

### 설계 요약
- 탄창의 스택 구조 규칙 자체를 변형하는 성장. 예: 중간 납탄 허용하되 그 아래 칸 봉인.
- "제약을 없애는 것이 아니라 바꾸는 방향"이 핵심 가드.

### 보류 사유
- 기본 LIFO 규칙의 재미를 먼저 검증해야 한다. 변형을 일찍 넣으면 기본 규칙의 긴장감이 희석될 위험.
- 입력 복잡도 상한(모바일 엄지 조작)도 고려 필요.

## 4. 납탄 위 명시적 자원 비용 추가

> **출처**: [11_core_mechanism_intention.md §2 설계 가드](file:///d:/ProjectLoB/docs/gdd/11_core_mechanism_intention.md)

### 설계 요약
- 현재 납탄 비용은 "템포 세금"(적 전진 + 한 턴 봉인) 단독.
- 여기에 행동력 같은 명시적 자원을 추가로 얹을지 보류 중.

### 보류 사유
- 문서 합의: "둘 다 걸면 납탄이 최후의 수단으로만 쓰여 메커니즘이 죽을 위험." v1은 템포 세금 단독으로 간다.
